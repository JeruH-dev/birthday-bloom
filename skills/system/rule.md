# 🌸 Development Rules

Developer: NABORAJ SARKAR

* Never break working systems.
* Never remove stable functionality unless explicitly instructed.
* Extend existing systems instead of rewriting them.
* Think before generating.
* Analyze the full architecture before making changes.
* Keep code scalable, modular, readable, and production-ready.
* Keep folder structure clean and maintainable.
* Avoid unnecessary dependencies, abstractions, and fake complexity.
* Write visually beautiful code with meaningful naming conventions.
* Comments must explain why the system exists and how it works.
* Maintain smooth, cinematic, emotionally engaging user experiences.
* Every animation, transition, and interaction must feel intentional.
* Maintain mobile-first responsive design and smooth 60fps performance.
* Optimize continuously for performance, scalability, SEO, accessibility, and maintainability.
* Prevent memory leaks, layout thrashing, duplicated logic, and unnecessary re-renders.
* Keep all public content SEO optimized and AI-crawler friendly.
* Protect security, APIs, environment variables, authentication systems, and user data.
* Sanitize and validate all external input.
* Whenever anything changes, update ALL related documentation.
* Keep README, feature docs, setup guides, architecture docs, and API docs synchronized.
* Never leave documentation outdated.
* Documentation must be visually organized, deeply explained, beginner-friendly, and emotionally engaging.
* Use diagrams, flows, screenshots, examples, and storytelling inside documentation whenever useful.
* Never generate robotic experiences.
* Everything must feel cinematic, immersive, emotional, premium, and intentional.
* Prioritize clarity, emotional quality, maintainability, scalability, and developer understanding.
* Never hallucinate APIs, fake systems, fake optimizations, or misleading architecture.
* Build experiences that feel alive, memorable, and beautifully engineered.
